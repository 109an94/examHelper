Chapter01 프로젝트 계획 [꿈꾸는 라이언]

  • 소프트웨어 생명 주기 (SDLC, Software Development Life Cycle)
  • 소프트웨어 개발 프로젝트의 시작부터 유지 보수까지의 전 과정을 정의한 구조로, 일반적으로 프로젝트 계획, 요구 분석, 설계, 구현, 테스트, 유지 보수의 6단계로 구성됩니다.
  • 폭포수 모형 (Waterfall Model)
  • 고전적, 전통적 개발 모형으로, 선형적이고 순차적으로 (Step-by-Step) 개발을 진행하며 이전 단계가 완료되어야 다음 단계로 넘어갈 수 있는 개발 모형입니다.
  • 프로토타입 모형 (Prototype Model)
  • 고객의 요구(need)를 파악하기 위해 견본이나 시제품을 만들어 최종 결과를 예측하고, 인터페이스 중심으로 요구사항 변경에 유연하게 대응할 수 있는 개발 모형입니다.
  • 나선형 모형 (Spiral Model)
  • 폭포수 모형과 프로토타입 모형에 위험 분석 기능을 추가하여 위험 관리 및 최소화를 중요시하며, 계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가 과정을 점진적으로 반복하는 개발 모형입니다.
  • 애자일 모형 (Agile Model)
  • 일정한 짧은 주기 (Sprint 또는 Iteration)를 반복하며 개발을 진행하여 고객 요구사항에 유연하게 대응하고 고객 소통/상호작용을 중시하는 개발 방법론입니다.
  • 하향식 설계 (Top-down Design)
  • 절차 지향적 방식으로, 최상위 컴포넌트를 먼저 설계한 후 하위 기능들을 점차적으로 부여하여 개발하며, 테스트를 초기부터 진행할 수 있는 설계 전략입니다.
  • 상향식 설계 (Bottom-up Design)
  • 객체 지향적 방식으로, 최하위 모듈을 먼저 설계한 후 이들을 결합하고 검사하며 시스템을 구축하는 설계 전략입니다.
  • 익스트림 프로그래밍 (eXtreme Programming, XP)
  • 애자일 방법론 중 하나로, 고객 참여와 신속한 개발 과정을 반복하며 용기, 단순성, 의사소통, 피드백, 존중 5가지 핵심 가치를 추구하고 테스트 주도 개발, 짝 프로그래밍 등을 기본 원리로 사용하는 개발 방법론입니다.
  • 전문가 감정 기법
  • 조직 내 두 명 이상의 전문가에게 비용 산정을 의뢰하는 하향식 비용 산정 기법으로, 개인적이고 주관적인 판단이 개입될 수 있습니다.
  • 델파이 기법 (Delphi Technique)
  • 한 명의 조정자와 여러 전문가의 의견을 종합하여 비용을 산정하는 기법으로, 전문가 감정 기법의 주관적 편견을 보완한 하향식 비용 산정 기법입니다.
  • LOC (Source Line of Code) 기법
  • 프로젝트 세부 작업 단위별로 비용을 정산하는 상향식 비용 산정 기법 중 하나로, 코드 라인 총 수를 기반으로 **낙관치(a), 비관치(b), 기대치(c)**를 측정/예측하여 비용을 산정합니다.
  • COCOMO (Constructive Cost Model)
  • 보헴(Boehm)이 제안한 원시 코드 라인 수를 기반으로 하며, 비용 견적의 유연성이 높아 널리 통용되는 수학적 비용 산정 기법입니다.
  • PUTNAM 모형
  • 소프트웨어 생명 주기 전 과정에 사용될 노력의 분포를 이용하여 비용을 산정하며 Rayleigh Norden 곡선의 노력 분포도를 기초로 하는 수학적 비용 산정 기법입니다.
  • SLIM
  • Rayleigh-Norden 곡선과 Putnam 모형을 기초로 개발된 자동화 추정 도구입니다.
  • Function Point (FP) 기법
  • 소프트웨어 기능 증대 요인 (자료 입력, 정보 출력, 명령어, 데이터 파일, 인터페이스)에 가중치를 부여 후 합산하여 기능 점수를 산출하는 수학적 비용 산정 기법입니다.
  • ESTIMACS
  • FP 모형을 기반으로 하여 개발된 자동화 추정 도구입니다.
  • WBS (Work Breakdown Structure)
  • 프로젝트 목표 달성을 위한 활동과 업무를 세분화하여 전체 프로젝트를 분할하고 수행 업무를 식별하여 일정 및 비용을 계획하는 기법입니다.
  • PERT (Program Evaluation and Review Technique)
  • 프로젝트 작업 상호관계를 네트워크로 표현하며, 불확실한 상황에 주로 사용되며 간선(화살표)에 각 작업별 낙관치/기대치/비관치를 기재하는 개발 일정 산정 기법입니다.
  • CPM (Critical Path Method)
  • 프로젝트 작업 상호관계를 네트워크로 표현하며 확실한 상황에 주로 사용되며, 노드(작업), 간선(의존 관계), 박스(이정표)로 구성되고, 경로상 가장 오래 걸리는 시간을 임계 경로로 파악하는 개발 일정 산정 기법입니다.
  • 간트 차트 (Gantt Chart)
  • 각 작업의 시작/종료 일정을 막대 바 (Bar) 도표를 이용하여 시간선(Time-line) 상에 표현하는 개발 일정 산정 도구입니다.
  • ① 프로젝트 계획 → ② 요구 분석 → ③ 설계 → ④ 구현 → ⑤ 테스트 → ⑥ 유지 보수
  • 소프트웨어 개발의 시작부터 유지 보수까지의 전 과정을 의 순서로 정의한 구조입니다. 생명주기

COCOMO 유형 관련 보충

  • 조직형 (Organic Mode)
  • 5만 라인 (50KDSI) 이하의 중·소규모 소프트웨어 개발 프로젝트에 적합하며, 팀원들이 프로젝트 경험이 풍부하고 유기적으로 협력하는 유형입니다.
  • 반분리형 (Semi-detached Mode)
  • 30만 라인 (300KDSI) 이하의 트랜잭션 처리 시스템 등 중간 규모의 소프트웨어에 적합하며, 팀원들이 프로젝트 경험이 혼합되어 있는 유형입니다.
  • 내장형 (Embedded Mode)
  • 30만 라인 (300KDSI) 이상의 최대형 규모 소프트웨어 관리 프로젝트에 적합하며, 하드웨어 및 소프트웨어 제약이 많아 복잡도가 높은 유형입니다.

임계 경로 (Critical Path) 관련 보충

  • 임계 경로 (Critical Path)
  • PERT/CPM 네트워크에서 프로젝트를 완료하는 데 가장 오랜 시간이 걸리는 경로를 의미하며, 프로젝트의 최소 완료 기간을 결정하는 경로입니다. 이 경로상의 작업이 지연되면 전체 프로젝트 완료 일정도 지연됩니다.
