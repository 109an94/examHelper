Chapter01 소프트웨어 설계 [꿈꾸는라이언]

  • 분할과 정복 (Divide and Conquer)
  • 하나의 큰 문제를 여러 개의 작은 서브 시스템으로 나누어 각각을 완성하는 방식으로, 복잡한 문제를 해결하는 데 사용되는 소프트웨어 설계 원리입니다.
  • 모듈화 (Modularity)
  • 시스템 기능을 모듈 단위로 분류하여 시스템의 성능 향상 및 재사용성을 높이는 소프트웨어 설계 원리입니다.
  • 추상화 (Abstraction)
  • 불필요한 부분은 생략하고 필요한 부분만 강조하여 모델화하며, 문제의 포괄적인 개념을 설계 후 차례로 세분화하여 구체화를 진행하는 설계 원리입니다.
  • 과정 추상화 (Procedural Abstraction)
  • 자세한 수행 과정을 정의하지 않고, 전반적인 흐름만 파악 가능하게 설계하는 추상화의 한 유형입니다.
  • 데이터(자료) 추상화 (Data Abstraction)
  • 데이터의 세부적 속성이나 용도를 정의하지 않고, 데이터 구조를 표현하는 추상화의 한 유형입니다.
  • 제어 추상화 (Control Abstraction)
  • 이벤트 발생의 정확한 절차나 방법을 정의하지 않고, 대표 가능한 표현으로 대체하는 추상화의 한 유형입니다.
  • 단계적 분해 (Stepwise Refinement)
  • 하향식 설계 전략으로, 추상화의 반복에 의한 세분화 과정을 거치며 세부 내역은 가능한 뒤로 미루어 진행하는 설계 기법입니다.
  • 정보 은닉 (Information Hiding)
  • 한 모듈 내부에 포함된 절차나 자료 관련 정보가 숨겨져 다른 모듈의 접근이나 변경이 불가능하게 하여 모듈을 독립적으로 수행할 수 있도록 해 유지보수를 용이하게 하는 설계 원리입니다.
  • Layer
  • 시스템을 계층으로 구분하고 구성하는 고전적 방식의 아키텍처 패턴입니다 (ex. OSI 참조 모델).
  • Client-server
  • 하나의 서버 컴포넌트와 다수의 클라이언트 컴포넌트로 구성되며, 클라이언트와 서버는 요청/응답을 제외하고 서로 독립적인 아키텍처 패턴입니다.
  • Pipe-Filter
  • 데이터 스트림 절차의 각 단계를 필터 컴포넌트로 캡슐화하여 데이터 전송, 재사용 및 확장성이 용이하며, 데이터가 단방향으로 흐르는 아키텍처 패턴입니다.
  • Model-view Controller (MVC)
  • 시스템 기능을 핵심 기능/데이터(모델), 사용자에게 정보 표시(뷰), 사용자 입력 처리(컨트롤러)로 개별 컴포넌트로 분리하여 대화형 애플리케이션에 적합한 아키텍처 패턴입니다.
  • Master-slave
  • 마스터 컴포넌트가 슬레이브 컴포넌트로 작업을 분할, 분리, 배포한 후 처리된 결과물을 다시 돌려받아 병렬 컴퓨팅에 사용되는 아키텍처 패턴입니다.
  • Broker
  • 컴포넌트와 사용자를 연결하여 분산 환경 시스템을 구축하는 데 사용되는 아키텍처 패턴입니다.
  • Peer-to-peer
  • 피어(Peer)를 하나의 컴포넌트로 산정하며, 각 피어는 클라이언트가 될 수도 서버가 될 수도 있는 (멀티 스레딩 방식) 아키텍처 패턴입니다.
  • Event-bus
  • 소스가 특정 채널에 이벤트 메시지를 발행하면 해당 채널을 구독한 리스너들이 메시지를 받아 이벤트를 처리하는 아키텍처 패턴입니다.
  • Blackboard
  • 컴포넌트들이 검색을 통해 공유 데이터 구조인 블랙보드에서 원하는 데이터를 찾아 상호작용하는 아키텍처 패턴입니다.
  • Interpreter
  • 특정 언어로 작성된 프로그램 코드를 해석하는 컴포넌트를 설계하는 데 사용되는 아키텍처 패턴입니다.
  • UML (Unified Modeling Language)
  • 고객과 개발자 간의 원활한 의사소통을 위해 표준화한 대표적인 객체 지향 모델링 언어입니다.
  • 인터페이스 (Interface)
  • 클래스나 컴포넌트가 구현해야 하는 오퍼레이션 세트를 정의하는 모델 요소입니다.
  • 사물 (Things)
  • UML 모델의 구성 요소 중 하나로, 구조(개념, 물리적 요소), 행동, 그룹, 주해(부가적 설명, 제약조건)를 포함합니다.
  • 연관관계 (Association)
  • 2개 이상의 사물이 서로 관련되어 있음을 표현하는 UML 관계입니다.
  • 집합관계 (Aggregation)
  • 하나의 사물이 다른 사물에 포함되는 (전체-부분) 관계를 표현하는 UML 관계입니다.
  • 포함관계 (Composition)
  • 집합 관계 내에서 한 사물의 변화가 다른 사물에게 영향을 미치는 (강한 전체-부분) 관계를 표현하는 UML 관계입니다.
  • 일반화관계 (Generalization)
  • 한 사물이 다른 사물에 비해 일반적이거나 구체적인지 표현하는 (한 클래스가 다른 클래스를 포함하는 상위 개념일 때) UML 관계입니다.
  • 의존관계 (Dependency)
  • 사물 간 서로에게 영향을 주는 관계로, 한 클래스가 다른 클래스의 기능을 사용할 때 주로 표현하는 UML 관계입니다.
  • 실체화관계 (Realization)
  • 한 객체가 다른 객체에게 오퍼레이션을 수행하도록 지정하거나 서로를 그룹화할 수 있는 관계를 표현하는 UML 관계입니다.
  • 클래스 다이어그램 (Class Diagram)
  • 클래스 사이의 관계 및 속성을 표현하는 UML 구조/정적 다이어그램입니다.
  • 객체 다이어그램 (Object Diagram)
  • 인스턴스를 객체와 객체 사이의 관계로 표현하는 UML 구조/정적 다이어그램입니다.
  • 컴포넌트 다이어그램 (Component Diagram)
  • 구현 모델인 컴포넌트 간의 관계를 표현하는 UML 구조/정적 다이어그램입니다.
  • 배치 다이어그램 (Deployment Diagram)
  • 물리적 요소(HW/SW)의 위치 및 구조를 표현하는 UML 구조/정적 다이어그램입니다.
  • 유스케이스 다이어그램 (Use case Diagram)
  • 사용자(Actor)와 사용 사례(Use Case)를 통해 사용자의 요구를 사용자 관점에서 분석하여 표현하는 UML 행위/동적 다이어그램입니다.
  • 시퀀스 다이어그램 (Sequence Diagram)
  • 시스템이나 객체들이 주고받는 메시지를 시간의 흐름에 따라 표현하는 UML 행위/동적 다이어그램입니다.
  • 커뮤니케이션 다이어그램 (Communication Diagram)
  • 객체들이 주고받는 메시지와 객체 간의 연관관계까지 함께 표현하는 UML 행위/동적 다이어그램입니다.
  • 활동 다이어그램 (Activity Diagram)
  • 객체의 처리 로직 및 조건에 따른 처리를 순서에 따라 표현하여 처리의 흐름을 보여주는 UML 행위/동적 다이어그램입니다.
  • CLI (Command Line Interface)
  • 텍스트 기반으로 명령어를 입력하여 시스템과 상호작용하는 사용자 인터페이스입니다.
  • GUI (Graphical User Interface)
  • 그래픽 기반으로 아이콘, 버튼 등을 사용하여 시스템과 상호작용하는 사용자 인터페이스입니다.
  • NUI (Natural User Interface)
  • 말이나 행동(터치, 제스처 등)을 통해 시스템과 상호작용하는 사용자 인터페이스입니다.
  • VUI (Voice User Interface)
  • 음성을 통해 시스템과 상호작용하는 사용자 인터페이스입니다.
  • OUI (Organic User Interface)
  • 사물과 사용자 간의 상호작용을 기반으로 하는 사용자 인터페이스입니다.
  • Wireframe
  • 기획 초기 단계에 대략적인 레이아웃을 설계하는 UI 설계 도구입니다.
  • Story Board
  • 와이어프레임, UI, 콘텐츠 구성, 프로세스 등을 포함하여 최종적인 산출문서 형태로 제작하는 UI 설계 도구입니다.
  • Prototype
  • 와이어프레임이나 스토리보드에 인터랙션(동적 효과)을 적용하여 실제 구현된 것처럼 테스트가 가능한 동적인 형태의 모형입니다.
  • Mockup
  • 실제 화면과 유사한 형태로 제작되지만 인터랙션이 적용되지 않은 정적인 형태의 모형입니다.
  • 인터랙션 (Interaction)
  • UI를 통해 시스템을 사용하는 일련의 상호작용, 즉 동적인 효과를 의미합니다.
