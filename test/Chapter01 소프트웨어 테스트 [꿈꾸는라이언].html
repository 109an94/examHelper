Chapter01 소프트웨어 테스트 [꿈꾸는 라이언]

  • 결함 집중 (Defect Clustering)
  • 파레토 법칙에 따라 전체 결함의 **80%**가 **20%**의 모듈에서 집중적으로 발생한다는 소프트웨어 테스트 기본 원리입니다.
  • 살충제 패러독스 (Pesticide Paradox)
  • 동일한 테스트 케이스에 의한 반복 테스트는 더 이상 새로운 버그를 발견할 수 없다는 소프트웨어 테스트 기본 원리입니다.
  • 오류-부재의 궤변 (Absence-of-errors Fallacy)
  • 소프트웨어에 결함이 없다 하더라도 사용자의 요구사항을 충족시키지 못하면 품질이 저하된다는 소프트웨어 테스트 기본 원리입니다.
  • 정적 테스트 (Static Test)
  • 프로그램을 실행하지 않고 명세서나 소스코드만을 분석하여 오류를 검사하는 테스트 방법입니다 (ex. 동료 검토, 워크 스루, 인스펙션).
  • 동적 테스트 (Dynamic Test)
  • 프로그램을 실제로 실행한 후 발생되는 오류를 검사하는 테스트 방법입니다 (ex. 화이트/블랙박스 테스트).
  • 명세 기반 테스트 (Specification-based Test)
  • 사용자의 요구사항에 대한 명세를 빠짐없이 테스트 케이스로 구현하는지 확인하는 테스트입니다 (ex. 동등 분할, 경계값 분석 등 블랙박스 테스트 기법).
  • 구조 기반 테스트 (Structure-based Test)
  • 소프트웨어의 내부 논리 흐름에 따라 테스트 케이스를 작성하고 확인하는 테스트입니다 (ex. 구문 기반, 결정 기반 등 화이트박스 테스트 기법).
  • 회복 테스트 (Recovery Test)
  • 시스템에 인위적인 결함을 부여한 후 정상으로 회복되는 과정을 확인하는 목적 기반 테스트입니다.
  • 안전 테스트 (Security Test)
  • 외부의 불법 침입으로부터 시스템을 보호할 수 있는지 확인하는 목적 기반 테스트입니다.
  • 강도 테스트 (Stress Test)
  • 과부하가 걸렸을 때 소프트웨어가 정상적으로 구동되는지 확인하는 목적 기반 테스트입니다.
  • 회귀 테스트 (Regression Test)
  • 소프트웨어 내 변경 또는 수정된 코드에 새로운 결함이 발생하지 않았음을 확인하는 목적 기반 테스트입니다.
  • 검증 (Verification)
  • 개발자의 시각에서 제품의 생산 과정을 테스트하며, 제품이 명세서대로 만들어졌는지 확인하는 것입니다 (ex. 단위/통합/시스템 테스트).
  • 확인 (Validation)
  • 사용자의 시각에서 생산된 제품의 결과를 테스트하며, 제품이 사용자의 요구사항을 충족하는지 확인하는 것입니다 (ex. 인수 테스트).
  • 화이트박스 테스트 (White Box Test)
  • 모듈 내부의 내용을 직접 볼 수 있으며, 내부의 논리적인 모든 경로를 테스트하여 소스코드의 모든 문장을 한 번 이상 수행하는 구조 기반 테스트입니다.
  • 구문(문장) 커버리지 (Statement Coverage)
  • 프로그램 내 모든 명령문을 적어도 한 번 이상 수행하는지 확인하는 화이트박스 테스트의 검증 기준입니다.
  • 결정(분기) 커버리지 (Branch Coverage)
  • 프로그램 내 전체 결정의 분기가 적어도 한 번씩은 실행되어 참/거짓 결과를 수행하는지 확인하는 화이트박스 테스트의 검증 기준입니다.
  • 조건 커버리지 (Condition Coverage)
  • 결정 명령문 내의 각 개별 조건식이 적어도 한 번은 참/거짓 결과를 수행하는지 확인하는 화이트박스 테스트의 검증 기준입니다.
  • 기초 경로 검사 (Base Path Testing)
  • 대표적인 화이트박스 테스트 기법으로, 테스트 케이스 설계자가 절차적 설계의 논리적 복잡성을 측정하고 실행 경로의 기초를 정의하는 지침으로 사용하는 테스트 기법입니다.
  • 블랙박스 테스트 (Black Box Test)
  • 모듈 내부의 내용을 알 수 없으며, 소프트웨어의 인터페이스에서 실시되어 각 기능이 완전히 작동되는 것을 입증하는 **'기능 테스트'**입니다.
  • 동치 분할 검사 (Equivalence Partition)
  • 프로그램 입력 조건에 타당한 입력 자료와 타당하지 않은 입력 자료의 개수를 균등하게 진행하여 테스트 케이스를 선정하는 블랙박스 테스트 기법입니다.
  • 경계값 분석 (Boundary Value Analysis)
  • 입력 조건의 경계값 (ex. 범위 구간의 양 끝: 이상/이하/초과/미만)을 테스트 케이스로 선정하는 블랙박스 테스트 기법입니다.
  • 오류 예측 검사 (Error Guessing)
  • 과거의 경험이나 확인자의 감각을 기반으로 잠재적 오류를 예측하고 테스트를 진행하는 블랙박스 테스트 기법입니다.
  • 동료 검토 (Peer Review)
  • 작성자가 내용 설명 후 동료들이 요구사항 명세서의 결함을 검토하는 요구사항 검증 기법입니다.
  • 워크 스루 (Walk through)
  • 요구사항 명세서를 미리 배포한 후 짧은 검토 회의를 진행하여 결함을 검토하는 요구사항 검증 기법입니다.
  • 인스펙션 (Inspection)
  • 작성자를 제외한 다른 전문가들이 요구사항 명세서의 결함을 검토하는 요구사항 검증 기법입니다.
  • 단위 테스트 (Unit Test)
  • 소프트웨어의 최소 단위 (모듈/컴포넌트)를 기반으로 진행되는 테스트로, 주로 구조 기반 테스트가 진행되며 기능성 테스트를 최우선으로 합니다.
  • 통합 테스트 (Integration Test)
  • 단위 테스트 후 모듈을 통합하는 과정에서 발생하는 오류 및 결함을 찾는 테스트 기법입니다.
  • 하향식 통합 테스트 (Top-down Integration Test)
  • 상위 모듈에서 하위 모듈 방향으로 통합하며, 하위 모듈의 기능을 단순히 수행하는 **스텁 (Stub)**을 사용하는 통합 테스트 기법입니다.
  • 상향식 통합 테스트 (Bottom-up Integration Test)
  • 하위 모듈에서 상위 모듈 방향으로 통합하며, 테스트 드라이버와 **클러스터 (Cluster)**를 사용하고 스텁(Stub)은 사용하지 않는 통합 테스트 기법입니다.
  • 시스템 테스트 (System Test)
  • 개발된 소프트웨어가 컴퓨터 시스템 내에서 작용하는지 점검하며, 실제 사용 환경과 유사한 테스트 환경에서 기능적 및 비기능적 테스트를 모두 수행합니다.
  • 인수 테스트 (Acceptance Test)
  • 사용자의 요구사항 충족 여부를 최종적으로 확인하는 테스트입니다.
  • 알파 테스트 (Alpha Test)
  • 인수 테스트의 일종으로, 통제된 환경에서 사용자가 개발자와 함께 확인하는 테스트입니다.
  • 베타 테스트 (Beta Test)
  • 인수 테스트의 일종으로, 통제되지 않은 환경에서 개발자 없이 여러 명의 사용자가 검증하는 테스트입니다.
  • 테스트 오라클 (Test Oracle)
  • 테스트 결과의 참/거짓을 판단하기 위해 사전에 정의된 참값을 대입하여 결과를 판정하는 도구입니다.
  • 테스트 드라이버 (Test Driver)
  • 테스트 대상의 하위 모듈을 호출하고 모듈 테스트 수행 후의 결과를 도출하는 도구로, 주로 상향식 통합 테스트에서 사용됩니다.
  • 테스트 스텁 (Test Stub)
  • 제어 모듈이 호출하는 하위 모듈의 역할을 단순 수행하는 도구로, 주로 하향식 통합 테스트에서 사용됩니다.
  • 테스트 슈트 (Test Suites)
  • 시스템에 사용되는 테스트 케이스의 집합 (컴포넌트/모듈)입니다.
  • 테스트 케이스 (Test Case)
  • 사용자의 요구사항 준수 여부 확인을 위해 설계된 테스트 항목 명세서 (입력값, 실행 조건, 기대 결과 등)입니다.
