Chapter01 소프트웨어 구현 [꿈꾸는라이언]

  • 소프트웨어 개발 프레임워크 (Software Development Framework)
  • 개발해야 할 애플리케이션 일부분이 이미 내장된 클래스 라이브러리에 구현되어 있어, 동일 로직 반복을 최소화하고 재사용성, 생산성, 유지보수성을 향상시키며 프레임워크가 애플리케이션 흐름을 제어하는 (제어의 역흐름) 기반 구조입니다.
  • 스프링 프레임워크 (Spring Framework)
  • JAVA 플랫폼을 위한 오픈소스 경량형 프레임워크로, 동적 웹사이트 개발과 전자정부 표준 프레임워크의 기반으로 널리 사용됩니다.
  • 전자정부 프레임워크
  • 우리나라 공공 부문 정보화 사업 시 효율적인 정보 시스템 구축을 지원하기 위해 필요한 기능과 아키텍처를 제공하는 프레임워크입니다.
  • 닷넷 프레임워크 (.NET Framework)
  • MS에서 개발한 Windows 프로그램 개발을 위한 프레임워크로, 공통 언어 런타임(CLR)이라는 가상 머신 상에서 작동합니다.
  • API (Application Programming Interface)
  • 소프트웨어 간의 인터페이스로, 서로 다른 소프트웨어/서비스 간 상호작용을 용이하게 하며 운영체제나 프로그래밍 언어의 기능을 프로그램에서 사용 가능하도록 구현한 것입니다.
  • Fan-in
  • 자신을 사용하는 모듈의 수입니다. 값이 높으면 재사용성 측면에서 설계가 우수하지만, 단일 장애점 발생 가능성이 높아 집중적인 관리와 테스트가 필요합니다.
  • Fan-out
  • 자신이 사용하는 모듈의 수입니다. 값이 높으면 불필요한 호출 가능성이 있어 단순화가 필요합니다.
  • 응집도 (Cohesion)
  • 개별 모듈이 독립적인 기능으로 정의되어 있는 정도로, 응집도가 높을수록 모듈의 품질이 향상됩니다.
  • 기능적 응집도 (Function Cohesion)
  • 모듈 내부의 모든 기능 요소가 단일 문제와 연관되어 수행되는 가장 응집도가 높은 형태입니다.
  • 순차적 응집도 (Sequential Cohesion)
  • 모듈 내 한 활동의 출력 데이터를 다음 활동의 입력 데이터로 사용하는 응집도 형태입니다.
  • 통신적 응집도 (Communication Cohesion)
  • 동일한 입/출력을 사용하여 서로 다른 기능을 수행하는 응집도 형태입니다.
  • 절차적 응집도 (Procedural Cohesion)
  • 모듈 내 구성 요소들이 다수 관련 기능을 순차적으로 수행하는 응집도 형태입니다.
  • 시간적 응집도 (Temporal Cohesion)
  • 특정 시간 내 처리되는 기능들을 모아 하나의 모듈로 작성하는 응집도 형태입니다.
  • 논리적 응집도 (Logical Cohesion)
  • 유사한 성격의 처리 요소들로 하나의 모듈이 형성되는 응집도 형태입니다.
  • 우연적 응집도 (Coincidental Cohesion)
  • 각 구성 요소들이 서로 관련 없는 요소로만 구성된, 가장 응집도가 낮은 형태입니다.
  • 결합도 (Coupling)
  • 개별 모듈 간 상호 의존하는 정도로, 결합도가 낮을수록 모듈의 품질이 향상됩니다.
  • 내용 결합도 (Content Coupling)
  • 한 모듈이 다른 모듈의 내부 기능 및 자료를 직접 참조/수정하는 가장 결합도가 높은 형태입니다.
  • 공통/공유 결합도 (Common/Share Coupling)
  • 공유되는 공통 데이터 (전역 변수 등)를 여러 모듈이 사용하여 상호 의존하는 결합도 형태입니다.
  • 외부 결합도 (External Coupling)
  • 한 모듈에서 선언한 데이터를 외부의 다른 모듈에서 참조하는 결합도 형태입니다.
  • 제어 결합도 (Control Coupling)
  • 한 모듈이 다른 모듈의 상세한 처리 절차를 알고 있어 처리 방식의 제어 요소도 함께 전달하는 결합도 형태입니다.
  • 스탬프 결합도 (Stamp Coupling)
  • 두 모듈이 동일한 자료 구조 (배열, 오브젝트)를 조회하여, 자료 구조 포맷 변화 시 조회하고 있는 모든 모듈에 영향을 주는 결합도 형태입니다.
  • 자료 결합도 (Data Coupling)
  • 모듈 간의 인터페이스가 자료 요소로만 구성되어, 가장 결합도가 낮은 형태입니다.
  • 모듈의 독립성 (Modularity Independence)
  • 모듈 간 과도한 상호작용을 막고 하나의 독립적 기능만을 수행하도록 하는 성질로, 높은 응집도와 낮은 결합도를 가질 때 높아집니다.
  • 객체 (Object)
  • 고유 식별자를 가지고 하나의 독립된 존재로, 상태(속성값)와 행위(연산, Method)로 정의되는 객체지향의 기본 요소입니다.
  • 클래스 (Class)
  • 공통 속성과 연산(행위)을 갖는 객체들의 집합이자 데이터 추상화 단위입니다.
  • 인스턴스 (Instance)
  • 클래스에 속한 각각의 객체를 의미합니다.
  • 추상화 (Abstraction - 객체지향)
  • 객체의 공통적인 속성과 기능을 추출하여 정의하고 불필요한 세부 내용은 숨겨, 유연한 설계와 유지보수가 용이하도록 하는 객체지향의 원리입니다.
  • 캡슐화 (Encapsulation)
  • 데이터와 데이터 처리 함수를 하나로 묶어 세부 내용을 외부 접근으로부터 숨기는 (정보 은닉) 객체지향의 원리입니다.
  • 상속 (Inheritance)
  • 상위 클래스의 속성과 연산을 하위 클래스가 물려받아 재사용성을 높이는 객체지향의 원리입니다.
  • 다형성 (Polymorphism)
  • 하나의 메시지에 각 객체별 고유 특성에 따라 여러 형태의 응답을 할 수 있도록 하는 객체지향의 원리입니다.
  • 오버라이딩 (Overriding)
  • 상위 클래스로부터 상속받은 메서드를 하위 클래스에서 메서드 이름, 매개변수, 반환 타입은 동일하게 유지하면서 재정의하는 다형성의 한 방법입니다.
  • 오버로딩 (Overloading)
  • 메서드 이름은 동일하나 매개변수 개수 또는 타입을 다르게 지정하여 여러 기능을 수행하도록 하는 다형성의 한 방법입니다.
  • 단일 책임 원칙 (SRP, Single Responsibility Principle)
  • 객체지향 설계 5대 원칙(SOLID) 중 하나로, 모든 클래스나 객체는 하나의 책임만을 가져야 하며 완전한 캡슐화를 지향해야 한다는 원칙입니다.
  • 개방 폐쇄의 원칙 (OCP, Open Closed Principle)
  • 객체지향 설계 5대 원칙(SOLID) 중 하나로, 소프트웨어 요소는 **확장에는 개방(Open)**되고 **수정에는 폐쇄(Close)**되어야 한다는 원칙입니다.
  • 리스코프 교체 원칙 (LSP, Liskov Substitution Principle)
  • 객체지향 설계 5대 원칙(SOLID) 중 하나로, 상위 클래스의 행동 규약을 하위 클래스가 위반하면 안 되며, 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다는 원칙입니다.
  • 인터페이스 분리 원칙 (ISP, Interface Segregation Principle)
  • 객체지향 설계 5대 원칙(SOLID) 중 하나로, 클라이언트가 비사용 메서드에 의존하지 않도록 인터페이스를 작게 분리해야 한다는 원칙입니다.
  • 의존성 역전 원칙 (DIP, Dependency Inversion Principle)
  • 객체지향 설계 5대 원칙(SOLID) 중 하나로, 의존관계 수립 시 변화하기 쉬운 구체적인 것 대신 변화하기 어려운 추상성이 높은 상위 클래스에 의존해야 한다는 원칙입니다.
  • 디자인 패턴 (Design Pattern)
  • 객체지향 프로그래밍 설계 시 자주 발생하는 문제에 대한 반복적 해결 방법을 정의한 참조 모델입니다.
  • Abstract Factory 패턴 (생성 패턴)
  • 구체적인 클래스에 의존하지 않고, 서로 연관되거나 의존적인 객체들이 조합된 인터페이스를 제공하는 생성 패턴입니다.
  • Builder 패턴 (생성 패턴)
  • 객체 생성 단계를 캡슐화하고 분리하여 객체를 조립하며 생성하고, 동일한 객체 생성 절차에서 서로 다른 표현 결과를 제공할 수 있는 생성 패턴입니다.
  • Factory Method 패턴 (생성 패턴)
  • 상위 클래스에서 객체 생성 인터페이스를 정의하지만, 실제 인스턴스를 만드는 클래스는 서브 클래스에서 결정하도록 분리하는 생성 패턴입니다.
  • Prototype 패턴 (생성 패턴)
  • 원본/원형 객체를 복제하는 방식으로 객체를 생성하는 생성 패턴입니다.
  • Singleton 패턴 (생성 패턴)
  • 클래스에서 하나의 객체만 생성 가능하며, 해당 객체를 어디서든 참조할 수 있도록 하는 생성 패턴입니다.
  • Adapter 패턴 (구조 패턴)
  • 비호환 인터페이스에 호환성을 부여하도록 변환하는 패턴으로, 기존 클래스를 감싸서 (Wrapper) 인터페이스를 변환합니다.
  • Bridge 패턴 (구조 패턴)
  • 구현부에서 추상층을 분리하여 각각 독립적으로 변형/확장 가능하게 하는 구조 패턴입니다.
  • Composite 패턴 (구조 패턴)
  • 트리 구조로 부분/전체 계층을 표현하며, 복합 객체와 단일 객체를 구분 없이 사용할 수 있도록 하는 구조 패턴입니다.
  • Decorator 패턴 (구조 패턴)
  • 상속 사용 없이 객체 간 결합을 통해 객체 기능을 동적으로 추가/확장하는 구조 패턴입니다.
  • Façade 패턴 (구조 패턴)
  • 상위에 인터페이스를 구성하여 서브 시스템의 복잡한 기능을 단순한 인터페이스로 구현하는 구조 패턴입니다.
  • Flyweight 패턴 (구조 패턴)
  • 인스턴스를 공유하여 메모리를 절약하고 클래스를 경량화하는 구조 패턴입니다.
  • Proxy 패턴 (구조 패턴)
  • 접근이 힘든 객체를 연결하는 인터페이스 역할 (대리 객체 수행)을 하는 구조 패턴입니다.
  • Chain of Responsibility 패턴 (행위 패턴)
  • 처리가능한 객체가 둘 이상 존재하여 한 객체 내에서 처리 불가 시 다음 객체로 이관하여 처리되도록 하는 행위 패턴입니다.
  • Command 패턴 (행위 패턴)
  • 요청 명령들을 추상/구체 클래스로 분리하여 단순화/캡슐화하는 행위 패턴입니다.
  • Interpreter 패턴 (행위 패턴)
  • 특정 언어에 문법 표현을 정의하는 행위 패턴입니다.
  • Iterator 패턴 (행위 패턴)
  • 컬렉션 객체의 내부 구조를 숨기고 요소들을 순차적으로 접근할 수 있도록 하는 행위 패턴입니다.
  • Mediator 패턴 (행위 패턴)
  • 객체들 간의 복잡한 상호작용을 캡슐화하여 별도의 객체로 정의한 후 중재하는 행위 패턴입니다.
  • Memento 패턴 (행위 패턴)
  • 객체를 이전의 특정 시점의 상태로 저장하고 복원하며 캡슐화를 유지하는 행위 패턴입니다.
  • Observer 패턴 (행위 패턴)
  • 한 객체의 상태 변화 시 상속되어 있는 다른 객체들에 변화를 전달하여 일관성을 유지하는 행위 패턴입니다.
  • State 패턴 (행위 패턴)
  • 객체의 상태에 따라 동일한 동작을 다르게 처리하도록 하는 행위 패턴입니다.
  • Strategy 패턴 (행위 패턴)
  • 동일 계열 알고리즘을 개별적으로 캡슐화하여 상호 교환할 수 있도록 하는 행위 패턴입니다.
  • Template Method 패턴 (행위 패턴)
  • 여러 클래스에서 공통 사용 메서드를 상위 클래스에서 정의하고, 하위 클래스마다 다르게 구현해야 하는 세부 사항을 개별 구현하도록 하는 행위 패턴입니다.
  • Visitor 패턴 (행위 패턴)
  • 각 클래스 데이터 구조로부터 처리/연산 기능을 분리하여 별도의 클래스를 만들고, 해당 클래스의 메서드가 각 클래스를 돌아다니며 특정 작업을 수행하도록 하여 객체 구조 변경 없이 새로운 연산 기능만 추가하는 행위 패턴입니다.
