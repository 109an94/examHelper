Chapter01 요구사항 분석 [꿈꾸는 라이언]

  • 요구사항 (Requirement)
  • 어떠한 문제를 해결하기 위해 필요한 조건 및 제약사항을 요구하는 것으로, 소프트웨어 개발 및 유지 보수 과정에 필요한 기준과 근거를 제공합니다.
  • 기능적 요구사항 (Functional Requirement)
  • 실제 시스템 수행에 필요한 기능과 관련된 요구사항입니다 (ex. 금융 시스템의 조회/인출/입금 기능).
  • 비기능적 요구사항 (Non-functional Requirement)
  • 성능, 보안, 품질, 안정성 등 실제 기능 수행에 보조적인 요구사항 또는 제약 사항입니다 (ex. 모든 화면이 3초 이내에 보여야 한다).
  • 요구사항 개발 프로세스 순서
  • ① 도출/추출 → ② 분석 → ③ 명세 → ④ 확인/검증의 순서로 진행되는, 소프트웨어 개발에 필요한 요구사항을 정의하고 구체화하는 일련의 과정입니다.
  • 도출/추출 (Elicitation)
  • 이해관계자들이 모여 요구사항을 식별하고 이해하는 과정으로, 인터뷰, 설문, 브레인스토밍, 유스케이스 등의 기법이 사용됩니다.
  • 분석 (Analysis)
  • 사용자 요구사항에 대한 타당성 조사를 수행하고, 비용 및 일정에 대한 제약 조건을 설정하는 단계입니다.
  • 명세 (Specification)
  • 요구사항을 체계적으로 분석한 후 승인 가능하도록 공식적인 형태로 문서화하는 단계입니다.
  • 확인/검증 (Validation)
  • 요구사항 명세서가 정확하고 완전하게 작성되었는지 검토하는 단계입니다.
  • CASE (Computer Aided SW Engineering)
  • 소프트웨어 개발 과정의 전부 또는 일부를 자동화하고 지원하는 도구 및 기법으로, 요구사항 분석을 돕는 도구로도 활용됩니다.
  • SADT (Structured Analysis and Design Technique)
  • SoftTech 사에서 개발한 요구사항 분석 도구로, 구조적 분석 및 설계 분석에 사용됩니다.
  • SREM (Software Requirements Engineering Methodology)
  • 실시간 처리 소프트웨어 시스템에서 요구사항을 명확하게 기술할 목적으로 개발된 요구사항 분석 도구입니다.
  • HIPO (Hierarchy Input Process Output)
  • 하향식 설계 방식에 사용되며, 가시적/총체적/세부적 다이어그램으로 구성되어 기능과 자료의 의존 관계를 동시 표현하여 이해가 쉽고 유지보수가 간단한 분석 도구입니다.
  • 자료 흐름도 (DFD, Data Flow Diagram)
  • 구조적 분석 기법에 이용되는 모델로, 프로세스 (원), 자료 흐름 (화살표), 자료 저장소 (평행선), 단말 (사각형)을 사용하여 시스템 내의 자료 흐름을 표현하며 버블(bubble) 차트라고도 불립니다.
  • 자료 사전 (DD, Data Dictionary)
  • 자료 흐름도(DFD)에 기재된 모든 자료의 상세 정의와 설명을 담고 있는 문서로, 정의(=), 택일([ ]), 반복({ }), 선택(( )), 구성(+), 설명()** 등의 표기법을 사용합니다.
  • Booch (부치)
  • 미시적 및 거시적 개발 프로세스를 모두 사용하며 클래스 및 객체 분석 및 식별에 중점을 둔 객체 지향 분석 모델입니다.
  • Jacobson (제이콥슨)
  • Use case를 사용하여 사용자나 외부 시스템이 시스템과 상호작용하는 방식을 중심으로 분석하는 객체 지향 분석 모델입니다.
  • Rumbaugh (럼바우)
  • 가장 일반적으로 사용되는 객체 지향 분석 모델로, **객체 모델링(Object), 동적 모델링(Dynamic), 기능 모델링(Function)**의 세 가지 모델로 구분하여 시스템을 분석합니다.
  • 정형 명세 (Formal Specification)
  • 수학적 원리를 기반으로 하여 정확하고 간결한 요구사항 표현이 가능하지만, 어려운 표기법으로 인해 사용자 이해가 어려운 요구사항 명세 기법입니다 (ex. VDM, Z, Petri-net).
  • 비정형 명세 (Informal Specification)
  • 자연어나 그림 중심으로 표현되어 의사소통은 용이하나, 작성자에 따라 모호한 내용으로 인해 일관성이 떨어질 수 있는 요구사항 명세 기법입니다.
  • 자료 흐름도 (DFD)에서 **원 (Circle/Bubble)**으로 표현되는 구성 요소
  • 프로세스 (Process). 자료를 변환하거나 처리하는 활동을 나타냅니다.
  • 자료 흐름도 (DFD)에서 **화살표 (Arrow)**로 표현되는 구성 요소
  • 자료 흐름 (Flow). 자료가 시스템의 한 구성 요소에서 다른 구성 요소로 이동하는 것을 나타냅니다.
  • 자료 흐름도 (DFD)에서 **평행선 (Parallel Lines)**으로 표현되는 구성 요소
  • 자료 저장소 (Data store). 자료가 저장되어 있는 곳 (파일, 데이터베이스 등)을 나타냅니다.
  • 자료 흐름도 (DFD)에서 **사각형 (Rectangle)**으로 표현되는 구성 요소
  • 단말/외부 개체 (Terminator). 시스템의 경계 밖에 존재하며 자료를 제공하거나 받는 개체(사용자, 외부 시스템 등)를 나타냅니다.

    자료 사전 (DD) 표기법 보충
  • 자료 사전(DD)에서 **등호( = )**로 표현되는 표기법
  • 정의. 자료 항목의 내용을 정의하는 것을 나타냅니다. (예: 주소 = 우편번호 + 상세주소)
  • 자료 사전(DD)에서 **대괄호 $[ ]**로 표현되는 표기법
  • 택일. 여러 개의 항목 중 하나를 선택하는 것을 나타냅니다. (예: 성별 [ 남성 | 여성 ])
  • 자료 사전(DD)에서 **소괄호 $( )**로 표현되는 표기법
  • 선택/생략 가능. 해당 자료 항목이 생략될 수도 있음을 나타냅니다.
  • 자료 사전(DD)에서 **중괄호 ${ }**로 표현되는 표기법
  • 반복. 해당 자료 항목이 여러 번 반복됨을 나타냅니다. (예: {상품명, 가격} 5 → 상품명과 가격이 5회 반복)
  • 자료 사전(DD)에서 **더하기 기호 +**로 표현되는 표기법
  • 구성. 여러 자료 항목이 합쳐져 하나의 자료 항목을 구성함을 나타냅니다.
  • 자료 사전(DD)에서 별표 * 로 표현되는 표기법
  • 설명/주석. 해당 자료 항목에 대한 부가적인 설명이나 주석을 나타냅니다.

  • ① 도출/추출 → ② 분석 → ③ 명세 → ④ 확인/검증
  • 소프트웨어 개발에 필요한 요구사항을 정의하고 구체화하는 일련의 과정입니다.
