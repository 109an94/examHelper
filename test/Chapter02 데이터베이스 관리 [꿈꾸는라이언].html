Chapter02 데이터베이스 관리 [꿈꾸는 라이언]

  • DBMS (DataBase Management System)
  • 특정 조직 내 필요한 데이터들의 모임인 데이터를 공용으로 소유, 유지, 이용할 수 있도록 관리하고 제어하는 시스템입니다.
  • 외부 스키마 (External Schema/Subschema)
  • 사용자 관점의 스키마로, 하나의 데이터베이스에 여러 개가 존재하며 사용자나 프로그램마다 다양한 형태의 논리적 구조로 나타납니다.
  • 개념 스키마 (Conceptual Schema)
  • 사용자와 DB 관리자 관점의 스키마로, 데이터베이스의 전체적인 논리적 구조를 명세하며 일반적으로 하나의 데이터베이스에 하나만 존재합니다.
  • 내부 스키마 (Internal Schema)
  • DB 설계자/개발자 관점의 스키마로, 개념 스키마를 물리적 저장 장치에 구현하는 방법, 즉 물리적 구조와 내부 레코드의 물리적 순서를 정의합니다.
  • 논리적 데이터 독립성 (Logical Data Independence)
  • 응용 프로그램에 영향을 주지 않고 데이터베이스의 논리 구조를 변경할 수 있는 성질입니다.
  • 물리적 데이터 독립성 (Physical Data Independence)
  • 응용 프로그램 및 논리 구조에 영향을 주지 않고 데이터의 물리적 구조를 변경할 수 있는 성질입니다.
  • DDL (Data Definition Language)
  • 데이터베이스의 구조(스키마)와 제약 조건을 정의하는 데 사용되는 언어입니다 (ex. CREATE, ALTER, DROP).
  • DML (Data Manipulation Language)
  • 데이터의 처리 및 조작에 사용되는 언어입니다 (ex. SELECT, INSERT, DELETE, UPDATE).
  • DCL (Data Control Language)
  • 데이터의 보안, 권한, 무결성, 병행 제어 등을 위해 사용되는 언어입니다 (ex. GRANT, REVOKE).
  • 데이터베이스 설계 순서
  • 데이터베이스 개발 과정을 순차적으로 진행하는 단계로, 요구 조건 분석 → 개념적 모델 → 논리적 모델 → 물리적 모델 → 구현 순으로 진행됩니다.
  • 개념적 모델링 단계
  • 현실 세계의 인식을 추상적 개념 세계로 표현하며, E-R 모델을 사용하여 독립적인 개념 스키마와 트랜잭션을 모델링하는 데이터베이스 설계 단계입니다.
  • 논리적 모델링 단계
  • 개념적 모델을 컴퓨터가 처리할 수 있는 구조로 표현하며, 관계 모델을 기반으로 정규화를 수행하고 논리적 스키마를 설계하는 단계입니다.
  • 물리적 모델링 단계
  • 데이터의 실제 저장 방법 및 접근 경로 (레코드 형식/순서, 인덱스 등)를 표현하며, 반정규화를 수행하고 목표 DBMS에 종속적인 물리적 구조로 변환하는 단계입니다.
  • 개체 (Entity)
  • 독립적이고 구별 가능하며 의미를 가지는 모든 것으로, 데이터 모델링의 기본 구성 요소입니다 (E-R 모델에서 사각형으로 표현).
  • 속성 (Attribute)
  • 개체가 가진 고유의 특성이자 가장 작은 논리적 단위입니다 (E-R 모델에서 원으로 표현).
  • 관계 (Relationship)
  • 두 개체 간의 의미 있는 연관성 및 연결 구조입니다 (E-R 모델에서 마름모로 표현).
  • 릴레이션 (Relation)
  • 관계형 데이터베이스에서 데이터를 표현하는 **표(Table)**를 의미합니다.
  • 튜플 (Tuple)
  • 릴레이션의 행에 해당하는 요소로, 레코드라고도 불리며 실제 데이터의 집합입니다.
  • 속성 (Attribute - 관계형 DB)
  • 릴레이션의 열에 해당하는 요소로, 필드라고도 불리며 릴레이션의 특성을 나타냅니다.
  • 차수 (Degree)
  • 릴레이션에 포함된 속성의 개수를 의미합니다.
  • 기수 (Cardinality)
  • 릴레이션에 포함된 튜플의 개수를 의미합니다.
  • 도메인 (Domain)
  • 하나의 속성이 취할 수 있는 원자값들의 집합입니다 (ex. 성별 속성의 도메인은 {남, 여}).
  • 후보키 (Candidate Key)
  • 유일성과 최소성을 모두 만족시키며, 기본키로 사용될 수 있는 속성 또는 속성들의 집합입니다.
  • 유일성 (Uniqueness)
  • 하나의 키 값으로 하나의 튜플을 유일하게 식별할 수 있는 성질입니다.
  • 최소성 (Minimality)
  • 모든 레코드를 식별하는 데 최소한의 속성으로만 구성된 성질입니다.
  • 기본키 (Primary Key)
  • 후보키 중에서 선택되어 릴레이션의 튜플을 식별하기 위해 반드시 필요하며, 중복된 값과 NULL 값을 가질 수 없는 키입니다.
  • 대체키 (Alternate Key)
  • 후보키가 둘 이상일 때, 기본키로 선택되지 않은 나머지 후보키들을 의미합니다.
  • 슈퍼키 (Super Key)
  • 한 릴레이션 내 속성들의 집합으로 구성된 키로, 유일성은 만족하지만 최소성은 만족하지 못할 수도 있는 키입니다.
  • 외래키 (Foreign Key)
  • 다른 릴레이션의 기본키를 참조하는 속성 또는 속성들의 집합으로, 릴레이션 간의 참조 관계를 나타냅니다.
  • 개체 무결성 (Entity Integrity)
  • 기본키를 구성하는 어떤 속성도 NULL 값이나 중복 값을 가질 수 없다는 제약 조건입니다.
  • 도메인/속성 무결성 (Domain/Attribute Integrity)
  • 릴레이션 내 튜플들이 각 속성의 도메인에 지정된 값만을 가져야 한다는 제약 조건입니다.
  • 참조 무결성 (Referential Integrity)
  • 외래키는 NULL 값이거나, 반드시 참조 릴레이션의 기본키 값과 동일해야 한다는 제약 조건입니다.
  • 임의 접근 통제 (DAC, Discretionary Access Control)
  • 사용자의 신원/신분에 따라 접근 권한을 부여하며, 데이터 소유자가 접근 통제 권한을 지정하고 GRANT/REVOKE 명령어를 사용하는 접근 통제 기술입니다.
  • 강제 접근 통제 (MAC, Mandatory Access Control)
  • 주체와 객체의 등급을 비교 후 시스템이 접근 권한을 부여하며, DB 객체별로 보안 등급을 부여하고 사용자별로 인가 등급을 부여하는 접근 통제 기술입니다.
  • 역할 기반 접근 통제 (RBAC, Role-based Access Control)
  • 사용자의 역할에 따라 접근 권한을 부여하며, 중앙 관리자가 이를 지정하여 다중 프로그래밍 환경에 최적화된 접근 통제 기술입니다.
  • 뷰 (View)
  • 기본 테이블에 기반을 둔 이름을 가지는 가상의 테이블로, 저장 장치 내에 논리적으로만 존재하며 데이터의 논리적 독립성과 보안성을 제공하지만, 독립적인 인덱스 보유가 불가능하고 삽입/삭제/갱신에 제약이 따릅니다.
  • 사용자 정의 무결성 (User-defined Integrity)
  • 속성값들이 사용자가 정의한 제약 조건에 만족해야 한다는 무결성 제약 조건입니다. (ex. 나이는 0보다 커야 한다).
  • NULL 무결성 (NULL Integrity)
  • 릴레이션의 특정 속성 값이 **NULL 값(허용되지 않은 값)**을 가질 수 없도록 제한하는 제약 조건입니다. (이는 기본키에 적용되는 개체 무결성과는 별개로, 일반 속성에 적용될 수 있습니다.)
